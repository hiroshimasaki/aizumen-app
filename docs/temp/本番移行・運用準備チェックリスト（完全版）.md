# 本番移行・運用準備チェックリスト（完全版）

本ドキュメントは、ご提示いただいた「AiZumen」特有の検証項目と、システム全般の非機能要件を統合し、さらに運用開始後に発生しうるリスクを想定した派生検証タスクを加えたものです。

## 1. 運用・保守体制の整備（インフラ・基盤）

システムが止まらないこと、止まってもすぐに戻せることを確実にします。

| カテゴリ | チェック項目 | 派生して確認すべきこと |
| :--- | :--- | :--- |
| **データ保護** | [ ] DBバックアップ (PITR) の有効化と設定確認 (Supabase Pro) | [ ] **リストア訓練**: 実際に過去の特定時点へデータを復元し、整合性が保たれるか手順を確認したか？ |
| **コスト管理** | [ ] Google Cloud 予算アラート設定 | [ ] **クォータ制限**: Gemini等のAPI利用制限（Rate Limit）に達した際の、アプリ側のエラーハンドリングとユーザーへの告知UIは適切か？ |
| **監視・通知** | [ ] 重大システムエラーの管理者通知設定 (Railway/Supabase) | [ ] **通知の静寂化**: 軽微なエラーで通知が埋もれないよう、重要度（Fatal/Error/Warn）に応じた通知先の振り分けはできているか？ |
| **ログ管理** | [ ] 監査ログ・解析済みファイルのアーカイブ/クリーニング方針決定 | [ ] **ストレージコスト**: 解析済みファイル（図面等）が増大した際の、S3/Supabase Storageのライフサイクルポリシー（古いファイルの自動移行）は検討したか？ |
| **不正対策** | [ ] 悪意ある大量登録（ボット等）への対策 (reCAPTCHA等) | [ ] **IP制限/レートリミット**: 同一IPからの短時間での大量リクエスト（スクレイピング等）を遮断する設定（WAF等）は入っているか？ |

## 2. リリース直前最終検証（アプリケーション・機能）

本番環境（Liveモード）特有の挙動と、ユーザー体験の最終確認です。

| カテゴリ | チェック項目 | 派生して確認すべきこと |
| :--- | :--- | :--- |
| **データ整合性** | [ ] 商用データの最終インポート確認<br>[ ] 既存ユーザーの権限設定が意図通りか | [ ] **マイグレーション後のデータ検証**: インポートした古い図面データが、新機能（AI解析等）でエラーにならず正しく読み込めるか？ |
| **決済フロー** | [ ] Stripe Liveモードでの決済〜クレジット付与の疎通確認 | [ ] **決済失敗/中断時の挙動**: カード拒否や3Dセキュア中断時に、クレジットが付与されず、かつ「再試行」を促す画面が表示されるか？ |
| **クレジット管理** | [ ] クレジット枯渇時のボタン不活性化・警告表示の挙動 | [ ] **同時実行制御**: 1クレジットしかない状態で、2つのタブから同時に解析ボタンを押した場合に、二重消費やマイナス残高にならないか？ |
| **AI機能・UI** | [ ] 一括解析モーダル：全ファイルでのプレビュー確認<br>[ ] 類似検索：サムネイル比較と流用フローの確認<br>[ ] AIボタンの色・注釈の統一感 | [ ] **解析タイムアウト**: 巨大な図面ファイルや大量の解析依頼時に、ブラウザがタイムアウトせずに「解析中」を維持できるか（Websocketやポーリングの安定性）？ |
| **ドメイン・SSL** | [ ] 独自ドメイン (aizumen.com) のSSLと全リンク正常性 | [ ] **wwwあり/なしの統一**: `www.aizumen.com` と `aizumen.com` のどちらかに正規化（301リダイレクト）されているか？ |
| **メンテナンス** | [ ] メンテナンスモードのON/OFF切り替えと影響範囲 | [ ] **実行中ユーザーへの配慮**: メンテONにした瞬間に解析中のユーザーがいた場合、データが壊れず、適切な案内が表示されるか？ |

## 3. セキュリティ・法務（非機能要件の補完）

SaaSとして公開する上で、法的に身を守り、信頼性を担保するための項目です。

| 項目 | 詳細 | 派生して確認すべきこと |
| :--- | :--- | :--- |
| **法務ドキュメント** | 利用規約、プライバシーポリシー、特定商取引法に基づく表記の最終確認 | **AI利用の明記**: ユーザーがアップロードした図面がAIの学習に使われないこと（API利用であれば通常は学習されませんが）を明記し、ユーザーの不安を払拭できているか？ |
| **Cookie同意** | Cookieポリシーの表示と同意取得 | **外部サービス連携**: StripeやGoogle Analytics等のCookie利用について適切に説明されているか？ |
| **パスワードポリシー** | ユーザーのパスワード強度設定 | **漏洩対策**: Supabase Authのパスワード設定が「最低8文字以上」などのセキュアな設定になっているか？ |
| **個人情報保護** | ユーザー情報のマスキングとアクセス権 | **管理者による閲覧制限**: 運営側（管理者）がユーザーの機密性の高い図面を不必要に閲覧できない仕組み、または閲覧ログが残る仕組みがあるか？ |

## 4. 運用開始直後のモニタリング計画（Day 1 タスク）

リリースして終わりではなく、最初の数時間をどう見守るかの計画です。

- [ ] **リアルタイムログ監視**: リリース直後1時間は Sentry / Logtail 等を常時監視し、予期せぬ 500 エラーが発生していないか確認する。
- [ ] **Stripe ダッシュボード監視**: 最初の数件の決済が「完了（Succeeded）」になり、Webhookが正常に飛んでいるか確認する。
- [ ] **ユーザーフィードバック窓口**: 問い合わせフォームが本番環境で動作し、管理者にメールが届くか最終確認する。
- [ ] **SNS/エゴサーチ**: サービス名での投稿をチェックし、ログインできない等の致命的な不具合報告がないか確認する。
