### 1-1. メンテナンスモードの実装と運用

システム停止時間を安全に管理するため、二段構えのメンテナンスモードを導入しています。

#### 1. 運用方法の使い分け
*   **DBによる制御 (推奨・通常運用)**:
    *   SU画面（サービス管理者画面）から On/Off を切り替えます。
    *   反映が即時（サーバー再起動不要）で、ユーザーへのメッセージも動的に変更可能です。
*   **環境変数による制御 (緊急・DB作業用)**:
    *   `MAINTENANCE_MODE=true` をインフラ側（Railway等）で設定します。OFFならfalse
    *   DBが停止している場合や、ログイン不可な状態でも確実に全アクセスを遮断できます。
    *   ※ 反映にはサーバーの再起動（リデプロイ）が必要です。

#### 2. 開発者用プレビュー (バイパス機能)
メンテナンスモード中でも、以下の手順で動作確認が可能です。
0.  ブラウザをシークレットモードで起動します。
1.  通常起動のブラウザでメンテナンスモードに入っていることを確認します。
2.  シークレットモードのブラウザで、URLの末尾に `?preview=[MAINTENANCE_BYPASS_TOKENの値]` を付けてアクセスします。
    *   例: `https://aizumen.com/?preview=dev-preview-token-2026`
    *   `[MAINTENANCE_BYPASS_TOKENの値]` は、`.env.local` または Railway の環境変数 `MAINTENANCE_BYPASS_TOKEN` で設定されている値です。aizumen-preview-token
3.  シークレットモードのブラウザにトークンが保存され、以降は通常の操作が可能になります。
4.  確認が終わったら、シークレットモードを閉じてください。

### 1-2. データ保護と監視の初期設定
*   **DB容量・Storage制限 (Supabase Pro)**: 
    *   **DB**: 上限 8GB (ベクトルデータ 20万件程度までは十分に収まる範囲)
    *   **Storage**: 上限 100GB (大量の図面PDFを保存可能)
*   **DBバックアップ (Supabase Pro)**: PITR（ポイントインタイムリカバリ）の有効化を確認。
*   **Google Cloud 予算アラート**: Gemini API等のコスト超過を防ぐため、予算閾値での通知を設定。
*   **重大エラー通知**: Railwayのデプロイ失敗や、サーバーの500エラーが管理者に即座に通知されるよう設定。

## 3. 本番リリース・DB更新の手順
安全に新機能をリリースし、検証環境からデータを同期するための標準手順です。

### 3-1. 正確な同期 (Go-live) 手順
1.  **メンテナンスモード ON**: 本番環境の `MAINTENANCE_MODE=true` を反映。
2.  **DB同期**: `pg_dump` または専用の同期ツールで、検証DBから本番DBへデータ（構造・レコード）を転送。
    *   **【重要】インデックスについて**: `drawing_tiles` テーブル（画像タイルやベクトル値）もそのまま移行されるため、本番環境移行後に**図面の一括再インデックス（`batch_reindex.js`）をやり直す必要はありません**。
3.  **Storage同期**: 検証Storageのファイルを本番Storageへコピー。
4.  **アプリのリデプロイ**: GitHubから最新コードを本番環境へ反映。
5.  **開発者最終確認**: バイパストークンを使用して本番URLでの疎通確認。
6.  **メンテナンスモード OFF**: `MAINTENANCE_MODE=false` に戻して公開。

### 3-2. ロールバック（切り戻し）手順
問題発生時は以下の順序で直前の状態に復旧します。
*   **アプリの差し戻し**: Railway/Vercelの履歴から一つ前の成功デプロイを再稼働（Re-deploy）。
*   **DBの差し戻し**: 用意しておいたバックアップテーブルからのデータ復元、または PITR による復旧。

### 3-3. DBスケールアップ・再起動時の注意点・処理の中断と再開
インデックス一括更新（`batch_reindex.js`）などの重いバックグラウンド処理が稼働中に、DBのCompute Size変更やPCの移動（出退勤）が必要になった場合の安全な手順です。
1. **処理の安全停止**: 大量の接続エラーや不完全なデータ生成を防ぐため、プログラムを一旦停止します（プロセス終了、または `Ctrl+C`）。
2. **DBの再起動・設定変更**: Supabase上でCompute Sizeの変更や再起動を実施したり、PCのネットワーク環境を切り替えます。
3. **処理の再開（リカバリ）**: 稼働が安定・再開可能になったら、以下のコマンドを実行します。
   ```bash
   # server ディレクトリで実行
   node batch_reindex.js
   ```
4. **レジューム機能について**: スクリプトは「レジューム（途中再開）」に対応しています。各図面に対して「すでにタイル（ベクトルデータ）が生成されているか」を事前に確認し、生成済みのものは自動的にスキップします。これにより、二重の処理時間をかけることなく、未処理の図面から安全に続きが開始されます。

### 5-2. AI利用状況（RPM/TPM/RPD）の監視
本番環境 (Vertex AI) と開発環境 (Google AI SDK) で監視場所が異なります。

*   **開発環境 (Google AI SDK)**: Google AI Studio (`aistudio.google.com/usage`)。サービス名「Generative Language API」。
*   **本番環境 (Vertex AI)**: Cloud Console「Vertex AI」>「割り当てとシステム上限」。
    *   `Online prediction requests per minute` (RPM)
    *   `Online prediction tokens per minute` (TPM)
*   *補足*: お支払いレポートをSKUでグループ化することでコストの詳細把握が可能。

### 5-3. 継続的な改善のベストプラクティス
*   **型変更の3段階**: `カラム追加 (Expand)` → `データ移行 (Migrate)` → `旧カラム削除 (Contract)` で無停止更新。
*   **Geminiプロンプトの管理**: スキーマ定義（DDL）をプロンプトに添え、Up/DownのSQLセットをGit管理。
*   **ストレージコスト管理**: Supabase Storageのライフサイクルポリシーを検討。
