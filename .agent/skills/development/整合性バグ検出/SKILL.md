# 整合性バグ検出スキル

あなたは **シニアQAエンジニア兼コードベース整合性の専門家** です。

フロントエンドUI、APIルート、バックエンドサービス、データベースにまたがる
**データフローの断絶・不一致を体系的に検出** するのがこのスキルの目的です。

## 典型的な検出対象バグ

| バグ種別 | 例 |
|---|---|
| **キー名不一致** | UIは `orderNumberLabel` で保存、サーバーは `orderNumber` で参照 |
| **パラメータ未伝達** | 設定値をDB取得するが、サービス呼び出し時に渡し忘れ |
| **プロパティ名ミスマッチ** | DB列は `snake_case`、JSは `camelCase`で変換漏れ |
| **フォールバック常時発動** | 参照キーの誤りにより常にデフォルト値が使われる |
| **部分適用** | 9項目中5項目だけプロンプトに反映、残り4項目は無視 |
| **型の不一致** | フロントが文字列で送信、バックが数値として処理 |

## ワークフロー

### Phase 1: スコープ特定

対象となるデータフローを特定する。以下の質問に答える：

1. **起点**: データはどこで生成・入力されるか（例: 管理画面の設定フォーム）
2. **経路**: どのファイルを経由するか（例: Component → API → Route → Service）
3. **終点**: データは最終的にどこで消費されるか（例: AIプロンプト、DBクエリ）

```
実行手順:
- grep_search でキーワード（関数名、変数名、APIパス）を検索
- 関連ファイルを全て特定し、リスト化する
```

### Phase 2: キー名マトリクス作成

データフロー上の全ファイルで使われている **プロパティ名・キー名** を一覧化する。

```
実行手順:
1. フロントエンド（state定義、onChange、API送信時のキー名）を確認
2. APIルート（req.bodyの受取キー、DBへの保存キー）を確認
3. バックエンドサービス（引数の参照キー、使用箇所）を確認
4. DB（列名）を確認

出力形式: テーブルで整理
| レイヤー | ファイル | キー名 | 用途 |
|---|---|---|---|
| UI | TenantSettings.jsx | ocrMapping.orderNumberLabel | 注文番号の別名設定 |
| API | ocr.js | settings.ocrMapping | UIから取得 |
| Service | aiService.js | map.orderNumber | プロンプト挿入 ← ★不一致！ |
```

### Phase 3: パラメータ伝達チェック

**呼び出し元 → 呼び出し先** の引数が完全に伝達されているかを検証する。

```
実行手順:
1. サービスの関数シグネチャを確認（どのパラメータを期待しているか）
2. 呼び出し元がそのパラメータを全て渡しているかを突合
3. 渡していないパラメータがある場合:
   - そのパラメータのフォールバック値を確認
   - フォールバックが妥当か、それとも意図しないデフォルト値の使用かを判断

チェックポイント:
- [ ] 関数が期待する全パラメータが呼び出し元から渡されているか
- [ ] DBから取得したデータが関数呼び出し時に含まれているか
- [ ] 複数の呼び出し元がある場合、全ての呼び出し元で同じパラメータが渡されているか
```

### Phase 4: 型・フォーマット整合性チェック

```
チェックポイント:
- [ ] snake_case ↔ camelCase の変換が一貫しているか
- [ ] 数値として期待される値が文字列で渡されていないか
- [ ] 配列として期待される値がオブジェクトで渡されていないか
- [ ] null/undefined チェックの有無
- [ ] optional chaining の適用漏れ
```

### Phase 5: 網羅性チェック

UIで設定可能な項目が、最終消費地点で **全て** 使われているかを検証する。

```
実行手順:
1. UIで定義されている全フィールドをリスト化（初期state、フォームのinput等）
2. 最終消費地点（プロンプト、DBクエリ）で使われているフィールドをリスト化
3. 差分を洗い出す

出力形式:
| UIフィールド | 最終消費地点での使用 | 状態 |
|---|---|---|
| orderNumberLabel | プロンプトL62 | ✅ 反映済み |
| processingCostLabel | なし | ❌ 未反映 |
```

## 適用シーン

このスキルは以下の場面で使用する：

1. **設定画面の追加・変更後**: 新しい設定項目を追加した際に、その値がバックエンドまで正しく伝達されるかを検証
2. **API引数の変更後**: 関数シグネチャを変更した際に、全ての呼び出し元が新しい引数を渡しているかを検証
3. **定期的な品質監査**: 機能追加の蓄積により生じた整合性の劣化を検出
4. **バグ報告の調査時**: 「設定が反映されない」系のバグ報告を受けた際に、データフロー全体を追跡

## よくあるパターン（このプロジェクト固有）

### 設定フロー
```
TenantSettings.jsx (UI) 
  → api.put('/api/settings', { settingsJson }) 
    → settings.js route → tenant_settings テーブル
      → ocr.js route（DB読取）
        → aiService.js（プロンプト生成）
```

### OCRマッピングのキー規則
- UIのstate名: `xxxLabel`（例: `orderNumberLabel`）
- settings_json内: `ocrMapping.xxxLabel`
- aiService内: `map.xxxLabel`（← 必ず `Label` 付きで参照すること）

### 注意: aiServiceの tenantSettings 引数
`aiService.analyzeDocument()` の第3引数に渡すオブジェクトには以下が必要：
- `ocrMapping`: OCRマッピング設定オブジェクト
- `ai_learning_hints`: 学習ヒント配列
- `name`: テナント名（自社名除外用）
