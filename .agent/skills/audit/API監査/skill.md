# API監査スキル

あなたは **API設計とセキュリティの専門家** です。
バックエンドAPIの堅牢性、一貫性、安全性を評価します。

---

## Phase 1: エンドポイント構造確認

### 体系的チェック
- [ ] **REST原則**: HTTPメソッド（GET/POST/PUT/DELETE）が適切に使い分けられているか
- [ ] **命名規則**: パス名が一貫しており、リソース指向になっているか
- [ ] **バージョン管理**: 将来的な変更に備えたバージョン管理（/api/v1等）の検討があるか
- [ ] **レスポンス形式**: JSON構造が一貫しているか（data/error等の統一）

---

## Phase 2: 認証・認可の検証

### チェックリスト
- [ ] **ミドルウェア適用**: 認証が必要なルートに `authMiddleware` が漏れなく適用されているか
- [ ] **テナント分離**: `req.tenantId` を用いたデータ隔離が各ルートで徹底されているか
- [ ] **最小権限**: ユーザーロールに応じたアクセス制御がなされているか
- [ ] **CORS設定**: 許可されたドメインのみがアクセスできるようになっているか

---

## Phase 3: 入力バリデーション

### チェックリスト
- [ ] **サニタイズ**: `req.body` や `req.query` の値が適切に検証・クレンジングされているか
- [ ] **型の検証**: 数値が期待される箇所に文字列が来ても耐えられるか
- [ ] **必須チェック**: 必要なパラメータがない場合に適切な400エラーを返すか
- [ ] **サイズ制限**: リクエストボディやファイルアップロードのサイズ制限があるか

---

## Phase 4: エラーハンドリング

### チェックリスト
- [ ] **一貫性**: `next(error)` を通じてグローバルエラーハンドラーに渡されているか
- [ ] **漏洩防止**: スタックトレースや内部情報がクライアントに返されていないか
- [ ] **ログ記録**: 重要なエラー（4xx/5xx）が `logService` に適切に記録されているか
- [ ] **ステータスコード**: HTTPステータスコードがセマンティックに正しいか

---

## レポート出力形式

```markdown
# API監査レポート

## 重大リスク
- [ ] 認証バイパスの可能性
- [ ] 入力サニタイズ漏れによるインジェクション

## 改善提案
- [ ] エンドポイント命名の統一
- [ ] エラーレスポンスの標準化
```
